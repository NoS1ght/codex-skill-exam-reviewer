第六章：高级主题

最小哈希与LSH · 数据流概念漂移 · Hadoop/Spark/RDD

核心知识点总结

一、哈希技术 — 相似文档查找

1. Shingling 算法（k-shingle）

  • 一个文档的 k-shingle（或 k-gram）是指连续 k 个字符一起出现的序列
  • 例：k=2, Doc1=abcab → 2-shingles = {ab, bc, ca}
  • 两个文档的相似性 = 它们的 k-shingles 集合的Jaccard 相似度
Jaccard 相似度：sim(C₁, C₂) = a / (a+b+c)
其中 a = 两者都为1的位数，b/c = 一1一0的位数

2. 最小哈希 (MinHashing) — 签名矩阵计算

核心思想：把每一列转换成更小的签名矩阵，使其能放入内存且保持相似性。
最小哈希定义：首次出现1的行号。
性质：理论上 sim(C₁,C₂) = Pr(h(C₁)=h(C₂))

签名矩阵计算步骤：

  1. 用多个随机哈希函数（即行排列）对初始矩阵各行重新编号
  2. 对每一列，找到首次出现1的新行号，填入签名矩阵
  3. 重复多次（如3次）得到多行签名
最小哈希三步：
✦ ① 计算签名矩阵 → ② 通过签名找相似 → ③ 验证是否真相似（可选）
✦ 签名矩阵比原始矩阵小得多，可放入内存

3. LSH 局部敏感哈希

  • 原因：一一比较所有文档对非常困难，需要快速匹配
  • 思想：将数据放入桶中，每个桶里的数据非常相似（不一一匹配）
  • 方法：将签名矩阵的列向量划分为 b 个 bands（块），每块哈希到桶中
  • 如果两列相似性很高（>0.5），至少有一个 band 全部映射到同一个桶中

二、数据流挖掘与概念漂移

数据流的特征

  • 持续依次到达的对象，无穷长
  • 有概念漂移（模式随时间发生漂移）
  • 挑战：单程处理、内存限制、低时间复杂度

概念漂移

概念漂移：预测分析和机器学习中，模型试图预测的目标变量的统计特性随时间以不可预见的方式改变。即 条件概率 P(Y|X) 发生变化。

检测概念漂移的两种方法

方法原理缺点
基于分布等时间窗口检测 P(C|X) 变化窗口大小难定、漂移检测慢、虚假漂移
基于错误率错误率突然变大→概念漂移对噪声敏感、难处理缓慢变化、依赖模型健壮性

数据流分类：快速决策树 (VFDT)

  • 动态构造决策树，利用小部分数据增量式构建
  • 新数据满足划分条件时分裂节点
  • 若错误率变多 → 发生概念漂移 → 替换旧节点

数据流聚类：微簇 (Micro-cluster)

簇特征 CF = (N, LS, SS)
N = 点的个数，LS = 横坐标之和，SS = 横坐标平方和
✦ 可计算中心（LS/N）和半径
✦ 支持动态增量式计算（有可加性、可减性）
✦ 为什么需要：数据连续不断进入，只需加新增点的数据

三、Hadoop / Spark 生态系统

Hadoop

  • 定义：软件框架，用于跨大型计算机集群对大型数据集进行分布式处理 (TB/PB级别)
  • 设计准则：分而治之 + ①自动并行化 ②容错和自动恢复 ③给用户提供接口

Hadoop 核心组件

组件功能关键概念
HDFS分布式文件系统（存储）分块存储，NameNode（元数据）+ DataNode（实际数据），心跳机制，挂掉后重建
MapReduce分布式计算框架Map + Reduce 两次操作，大量IO
ZooKeeper协调服务集群管理和协调

Spark vs MapReduce

MapReduceSpark
适用于一次操作，多次操作不适用统一引擎，支持多种应用
共享机制落后，大量时间IO拥有 RDD 支持内存计算

RDD（弹性分布式数据集）

RDD：只读，便于内存计算
✦ Transformation（转换）：从现有数据集创建新数据集，懒惰执行（不立即计算）
✦ Action（动作）：运行计算后向驱动程序返回值（触发实际计算）

—— 选择题 ——

Q1最小哈希 (MinHash) 的定义是（ ）。单选
A. 最小行号的值
B. 首次出现1的行号
C. 最小列号的值
D. 最后出现1的行号
✓ B
解析：最小哈希 = 在某个行排列下，首次出现1的行号。对每一列独立计算，用多个哈希函数（行排列）得到签名矩阵的多行。
Q2LSH 局部敏感哈希中，将签名矩阵列向量划分为 b 个 bands，其原理是（ ）。单选
A. 每个band独立计算相似度
B. bands越多准确率越高
C. 相似列至少有一个band全部映射到同一个桶
D. band划分越多速度越快
✓ C
解析：LSH核心原理：相似度较大的列，至少有一个band内的所有行哈希值相同→映射到同一个桶。如果相似性>0.5，高概率至少一个band全匹配。
Q3概念漂移 (Concept Drift) 指的是（ ）。单选
A. 数据量的持续增长
B. 数据类型的多样化
C. 数据源的不可靠性
D. 目标变量的统计特性随时间不可预见地改变
✓ D
解析：概念漂移 = P(Y|X) 条件概率随时间改变。例如用户购买习惯随时间变化。A是Volume，B是Variety。
Q4基于错误率检测概念漂移的优点和缺点是（ ）。单选
A. 简单直观，但对噪声敏感
B. 对噪声鲁棒，但计算量大
C. 准确度高，但需要大量内存
D. 速度快，但无法检测缓慢漂移
✓ A
解析：基于错误率的方法有三大缺点：①对噪声敏感 ②对缓慢变化难以处理 ③取决于学习模型健壮性。A正确描述了其核心缺点。
Q5数据流聚类中，微簇的簇特征 CF 公式为（ ）。单选
A. CF = (N, Mean, Variance)
B. CF = (N, LS, SS) N点数/LS横坐标和/SS平方和
C. CF = (K, Center, Radius)
D. CF = (Count, Min, Max)
✓ B
解析：CF = (N, LS, SS)。N=点数，LS=横坐标和，SS=横坐标平方和。具有可加性和可减性，支持动态增量式计算，每个新数据只需加到CF中。
Q6Hadoop 中负责存储数据元信息（哪一块在哪个机器）的组件是（ ）。单选
A. MapReduce
B. ZooKeeper
C. NameNode
D. DataNode
✓ C
解析：NameNode 存储元数据（哪块数据在哪个机器），DataNode 存储实际数据。DataNode 有心跳机制（定时汇报），超时未汇报则视为宕机，自动重建。
Q7Spark 相比 MapReduce 的核心优势是（ ）。单选
A. RDD 支持内存计算，减少 IO 开销
B. 不需要分布式集群
C. 能处理更大规模的数据
D. 不需要编程
✓ A
解析：Spark 核心优势：RDD 支持内存计算，避免 MapReduce 大量磁盘IO。MapReduce每次操作都要读写磁盘，多次操作效率低。
Q8RDD 的两种操作方式中，Transformation 的特点是（ ）。单选
A. 立即计算并返回结果
B. 只能对数据做过滤
C. 一次使用后 RDD 即被销毁
D. 懒惰执行，不立即计算
✓ D
解析：Transformation（如map/filter/groupBy）是懒惰的，不立即计算，只记录操作链。直到遇到Action（如count/collect/save）才触发实际计算。
Q9Hadoop 的设计准则核心思想是（ ）。单选
A. 单机高性能计算
B. 分而治之，大量廉价机器并行计算
C. 使用GPU加速计算
D. 所有数据放入内存计算
✓ B
解析：Hadoop核心思想：分而治之，使用大量廉价机器并行计算，配合自动并行化、容错恢复和用户接口。B是Spark的特点（RDD内存计算），不是Hadoop的。
Q10Jaccard 相似度 sim(C₁, C₂) 的计算公式为（ ）。单选
A. (a+c) / (a+b+c)
B. (a+b) / (a+b+c)
C. a / (a+b+c)
D. b / (a+b+c)
✓ C
解析：Jaccard = a/(a+b+c)，其中 a=两者都为1的列数，b/c=一1一0的列数。分子是共同为1的数量，分母是至少有一个为1的总数。

FIB_1最小哈希 (MinHash) 的目的是把原始特征矩阵转换为更小的__________，使其能放入内存并进行相似度比较。填空
签名矩阵 (Signature Matrix)
解析：签名矩阵比原始矩阵小得多，但能保持原始矩阵的Jaccard相似度（理论上两者相等）。
FIB_2数据流的四个主要挑战是：单程处理、内存限制、低时间复杂度、__________。填空
概念漂移
解析：数据流四大挑战：单程处理（不能回头）、内存限制（不能全存）、低时间复杂度（快速处理）、概念漂移（模式随时间改变）。
FIB_3检测概念漂移的两种方法是：基于__________的方法和基于错误率的方法。填空
分布
解析：基于分布：在等时间窗口检测P(C|X)的变化。基于错误率：错误率突然变大即检测到漂移。各有优缺点。
FIB_4Hadoop 核心两大组件是：HDFS（分布式存储）和__________（分布式计算）。填空
MapReduce
解析：HDFS负责存储（NameNode+DataNode），MapReduce负责计算（Map+Reduce）。ZooKeeper负责协调。
FIB_5RDD操作中，__________是懒惰执行的（不立即计算），而 Action 触发实际计算并返回结果。填空
Transformation
解析：Transformation（map/filter等）只是记录操作，不计算。Action（count/collect/save等）才触发DAG执行，返回结果。
